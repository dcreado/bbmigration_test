FastFed Profile for SCIM Provisioning - draft 01
fastfed-1_0

Abstract

This specification defines the requirements to implement the FastFed Profile for SCIM Full Lifecycle provisioning. This profile supports continual provisioning, update, and deprovisioning of end-users between the Identity Provider and Application Provider.

1. Introduction

This specification defines the functionality which a provider must implement to satisfy the FastFed Profile for SCIM Full Lifecycle provisioning.

It consists of the following extensions to the FastFed Core specification:

  • Additional metadata in the FastFed Handshake Flows to exchange SCIM provisioning endpoints.
  • An interoperability profile describing the subset of the SCIM specifications which must be implemented to be FastFed Compatible for SCIM Full Lifecycle provisioning.

1.1. Requirements Notation and Conventions

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

In the .txt version of this specification, values are quoted to indicate that they are to be taken literally. When using these values in protocol messages, the quotes MUST NOT be used as part of the value. In the HTML version of this specification, values to be taken literally are indicated by the use of this fixed-width font.

1.2. Terminology

This FastFed Profile uses the terminology defined in Section 1.2 of the FastFed Core specification.

2. FastFed Metadata

This specification extends the FastFed Core metadata (Section 3.3.1) with the following value for provisioning_profiles:

urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle
A Provider who includes this URN in their list of capabilities MUST comply with the requirements described in this specification.

The following is a non-normative example of Provider Metadata showing the usage of the value: 

 {
   "identity_provider": {
     "capabilities": {
       "provisioning_profiles": ["urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle"],
   ...
 }

3. FastFed Handshake

When using this SCIM provisioning profile, the FastFed Identity Provider and Application Provider have various responsibilities to comply with the protocol.

The Identity Provider fulfills the role of a SCIM Client. It has a responsibility to provision user information into the Application Provider as described in Section 4.1.

The Application Provider fulfills the role of a SCIM Service. It has a responsibility to host a SCIM Endpoint and handle provisioning messages from the Identity Provider as described in described in Section 4.2

This specification extends the FastFed Core handshake messages with the additional attributes necessary for each party to fulfill their respective obligations under SCIM.

3.1. FastFed Registration Request

This specification extends the contents of the FastFed Registration Request (Section 7.2.3.1 of FastFed Core) with a structure sharing the same name as the profile: "urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle".

The structure contains the following attributes:

provider_identity
REQUIRED. A structure containing the contact information and authentication keys necessary to authenticate the provisioning provider, as defined in Section 3.3.3 of FastFed Core.

The provider_identity MAY be the same party who hosted the FastFed Handshake (and is defined in the Identity Provider FastFed Metadata). Alternatively, SCIM provisioning may be delegated to a distinct sub-system which authenticates with its own identity and key materials. By specifying a protocol-specific provider_identity for SCIM provisioning, both scenarios are supported.

The following is a non-normative example of the contents of a registration request message: 

 {
   "iss": "https://idp.example.com",
   "sub": "tenant-12345",
   "aud": "https://app.example.com",
   "exp": 1234567890,
   "schema": "urn:ietf:params:scim:schemas:core:2.0:User",
   "provisioning_profiles": [
     "urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle"
   ],
   "urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle": {
     "provider_identity": {
       "jwks_uri": "https://provisioning.example.com/keys",
       "contact_information": {
         "organization": "Example Inc.",
         "phone": "+1-800-555-5555",
         "email": "support@example.com"
       }
     }
   }
 }

3.2. FastFed Registration Response

This specification extends the contents of the FastFed Registration Response (Section 7.2.3.3 of FastFed Core) with a structure sharing the same name as the profile: "urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle".

The structure contains the following attributes:

scim_service_uri
REQUIRED. Contains the URL of the Application Provider's SCIM Endpoint.
oauth_metadata
REQUIRED. A structure containing the OAuth metadata for authenticating to the SCIM endpoint, as defined in Section 6.6 of FastFed Core.

The OAuth scope returned in this structure MUST authorize the SCIM client to perform all provisioning activity specified by this profile.

The following is a non-normative example of the contents of a registration response message: 

 {
   "urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle": {
     "scim_service_uri": "https://tenant-56789.app.example.com/scim"
     "oauth_metadata": { 
       "grant_type": "urn:ietf:params:oauth:grant-type:jwt-bearer",
       "token_endpoint": "https://tenant-56789.app.example.com/oauth",
       "scope": "scim"
     }
   }
 }

4. Interoperability Requirements

Each identity standard defines a set of optional features to enable usage in a wide variety of circumstances.

However, a consequence of the flexibility is that two Providers may find themselves incompabitible despite sharing the same protocols.

To deliver the simplified experience that is the goal of FastFed, it is important that two FastFed-enabled Providers have confidence that they can interoperate when sharing the same protocols.

The following sections describe the subset of the SCIM Protocol specifications that Providers MUST implement to be FastFed Compatible for this profile. Providers MAY support additional functionality, but MUST NOT require the additional functionality when configuring federation with another Provider using the FastFed specifications.

4.1. Identity Provider Requirements

SCIM requirements for Identity Providers:

  • MUST implement the required functionality of a SCIM Client as defined in [RFC7643] and [RFC7644].
  • MUST authenticate to the SCIM endpoint using an OAuth access_token. The token must be attained using the mechanisms defined in Section 6.6 of the FastFed Core specification. The keyId used to attain the OAuth token must exist in the keys hosted at the jwks_uri specified by the SCIM provider_authentication in Section 3.1.
  • For each end-user that is authorized to access the Application:
    • SHOULD replicate end-user information to the Application Provider within 60 minutes of an active end-user being created or updated.

      The Identity Provider MAY use any combination of the following operations to provision user information:
      • Create account via POST /Users
      • Update account details via PUT /Users/{id}
      • Read list of accounts via GET /Users
      • Read account details via GET /Users/{id}
      • Filtering based on "userName eq"

    • SHOULD replicate end-user deactivation to the Application within 15 minutes of the user being deactivated within the Identity Provider.

      The Identity Provider MAY use any combination of the following operations to deactivate users in the Application:
      • Deactivate account via PATCH /Users{id}
      • Read list of accounts via GET /Users
      • Read account details via GET /Users/{id}
      • Filtering based on "userName eq"

    • SHOULD populuate the groups attribute on a SCIM User object with a list of groups to which the user belongs. As specified in the SCIM Core Schema [RFC7643], a user can belong to a group either through direct membership, through nested groups, or dynamically calculated.

4.2. Application Provider Requirements

SCIM requirements for Application Providers:

  • MUST implement the required functionality of a SCIM Service Provider as defined in [RFC7643] and [RFC7644].
  • MUST support authentication via OAuth access_tokens.
  • MUST provide an OAuth authorization server capable of issuing tokens as defined in Section 6.6 of the FastFed Core specification. OAuth tokens MUST only be issued if the token request is signed using a keyId which exists in the key set hosted at the jwks_uri specified by the SCIM provider_authentication in Section 3.1.
  • MUST support the following operations:
    • Create account via POST /Users
    • Update account details via PUT /Users/{id}
    • Deactivate account via PATCH /Users{id}
    • Read list of accounts via GET /Users
    • Read account details via GET /Users/{id}
    • Filtering based on "userName eq"

  • SHOULD respond to account deactivation by revoking the ability for the end-user to use the Application, even if that end-user has an active session with an expiration date in the future. The revocation mechanism is an implementation detail and outside the scope of this specification. Revocation should occur within 15 minutes of receiving the deactivation.

5. Authentication Example

The following examples illustrate how an Identity Provider authenticates to a SCIM endpoint to perform SCIM provisioning activities.

The examples below adhere to the authentication requirements defined in Section 6 of the FastFed Core specification ("Provider Authentication Methods"), as well as the additional SCIM profile requirements defined in Section 3.1 and Section 3.2 of this specification.

5.1. Identity Provider Shares Public Key

As a first-step, the Identity Provider shares a public key with the Application Provider in the form of a JSON Web key Set [JWK]. This occurs during the FastFed Handhshake (see "jwks_uri", below) and is described in Section 3.1 of this specification. 

 {
   "iss": "https://idp.example.com",
   "sub": "tenant-12345",
   "aud": "https://app.example.com",
   "exp": 1234567890,
   "schema": "urn:ietf:params:scim:schemas:core:2.0:User",
   "provisioning_profiles": [
     "urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle"
   ],
   "urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle": {
     "provider_identity": {
       "jwks_uri": "https://provisioning.example.com/keys",
       "contact_information": {
         "organization": "Example Inc.",
         "phone": "+1-800-555-5555",
         "email": "support@example.com"
       }
     }
   }
 }

Upon receiving this information, the Application Provider may internally perform an OAuth client registration in order to capture the provider's contact information and jwks_uri, and authorize the Provider to perform SCIM provisioning activities. This is an implementation detail. FastFed does not require the exchange of an OAuth clientID.

5.2. Application Provider Shares SCIM and OAuth Endpoints

Subsequently in the FastFed Handshake, the Application Provider returns SCIM and OAuth metadata back to the Identity Provider. This is described in Section 3.2 of this specification. 

 {
   "urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle": {
     "scim_service_uri": "https://tenant-56789.app.example.com/scim"
     "oauth_metadata": {
       "grant_type": "urn:ietf:params:oauth:grant-type:jwt-bearer",
       "token_endpoint": "https://tenant-56789.app.example.com/oauth",
       "scope": "scim"
     }
   }
 }

5.3. Identity Provider Acquires an OAuth Access Token

After the FastFed Handshake is complete and the Identity Provider is ready to begin calling the SCIM endpoint, it must first acquire an OAuth access_token. This will be used to authenticate to the SCIM endpoint.

The access_token is acquired by using the JWT Authorization Grant Type. This grant type is defined in RFC 7523, and its usage within FastFed is specified in Section 6.6 of the FastFed Core specification.

In essence, the Identity Provider sends a signed JWT to the OAuth endpoint hosted by the Application Provider. If the JWT is valid and is signed by an authorized party, then an OAuth access_token is returned.

This is illustrated in the following steps. First, a JWT is constructed with the following contents: 

 {                                                                                                                 
   "iss": "idp.example.com",                                                                                       
   "sub": "tenant-12345",                                                                                          
   "aud": "app.example.com",                                                                                       
   "exp": 1234567890,            
 }

Next, a request is made to the OAuth token endpoint, using the signed and serialized JWT as the value of the assertion. 

 {
     POST /oauth HTTP/1.1
     Host: tenant-56789.app.example.com
     Content-Type: application/x-www-form-urlencoded

     grant_type=urn%3Aietf%3Aparams%3Aoauth%3Agrant-type%3Ajwt-bearer
     &assertion=eyJhbGciOiJFUzI1NiIsImtpZCI6IjE2In0.
     eyJpc3Mi[...omitted for brevity...].
     J9l-ZhwP[...omitted for brevity...]
 }

In response, an OAuth token is returned: 

     HTTP/1.1 200 OK
     Content-Type: application/json;charset=UTF-8
     Cache-Control: no-store
     Pragma: no-cache

     {
       "access_token":"2YotnFZFEjr1zCsicMWpAA",
       "token_type":"Bearer",
       "expires_in":3600
     }

The token is used to authenticate to the SCIM endpoint: 

   POST scim/Users  HTTP/1.1
   Host: tenant-56789.app.example.com
   Accept: application/scim+json
   Content-Type: application/scim+json
   Authorization: Bearer 2YotnFZFEjr1zCsicMWpAA
   Content-Length: ...

   {
     "schemas":["urn:ietf:params:scim:schemas:core:2.0:User"],
     "userName":"bjensen",
     "externalId":"bjensen",
     "name":{
       "formatted":"Ms. Barbara J Jensen III",
       "familyName":"Jensen",
       "givenName":"Barbara"
     }

When the access token expires, the SCIM service returns an HTTP 401 response code. The JWT Authorization Grant flow may then be re-executed to acquire a new access token.
