1.  Introduction

   This specification defines the functionality which a provider must
   satisfy to describe themselves as FastFed Compatible for the OpenID
   Connect protocol.

   It consists of the following extensions to the FastFed Core
   [FastFed.Core] specification:

   o  Additional metadata in the FastFed Handshake Flows in order to
      exchange OpenID Connect Discovery Endpoints.  tokens.

   o  Mapping rules for populating OpenID Connect claims from a SCIM
      schema.

   o  An interoperability profile describing the subset of the OpenID
      Connect specifications which must be implemented in order to be
      FastFed Compatible.

   o  The mechanics of key rotation for OpenID Connect.

1.1.  Requirements Notation and Conventions

   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
   "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and
   "OPTIONAL" in this document are to be interpreted as described in RFC
   2119 [RFC2119].

   In the .txt version of this specification, values are quoted to
   indicate that they are to be taken literally.  When using these
   values in protocol messages, the quotes MUST NOT be used as part of
   the value.  In the HTML version of this specification, values to be
   taken literally are indicated by the use of "this fixed-width font".

   This specification extends the FastFed Core [FastFed.Core] metadata
   (Section 3.3.1) with the following value for

      "sso_protocols_supported" indicates they comply with the
      requirements described in this specification.

3.  FastFed Handshake

   When using the OpenID Connect protocol, the FastFed Identity Provider
   and Application Provider have various responsibilities to comply with
   the protocol.

   The Identity Provider fulfills the role of an OpenID Provider.  It
   has a responsibility to support OpenID Connect Discovery and Dynamic
   Client Registration as described in Section 5.1.

   The Application Provider fulfills the role of an OpenID Relying
   Party.  It has a responsibility to perform Dynamic Registration with
   the provider, as described in Section 5.2

   This specification extends the FastFed Core [FastFed.Core] handshake
   messages with additional attributes necessary for each party to
   fulfill their respective obligations under OpenID Connect.

3.1.  FastFed Registration Request

   This specification extends the JWT contents of the FastFed
   Registration Request (Section 6.2.3.1 of FastFord Core
   [FastFed.Core]) with the following attributes:

   oidc_configuration_uri  Specifies the Issuer "location" for OpenID
      Provider Configuration Information, as defined in Section 4 of
      OpenID Connect Discovery 1.0 [OpenID.Discovery].

   oidc_initial_access_token  Specifies the "Initial Access Token" for a
      Client Registration Endpoint, as defined in Section 3 of OpenID
      Connect Dynamic Client Registration 1.0 [OpenID.Registration].

   The following is a non-normative example of the contents of a
   registration request message prior to serialization:

4.  Schema Binding

   FastFed recommmends SCIM for user schemas.

   This specification defines how to be FastFed Compliant when using
   SCIM schemas and endpoints with the OpenID Connect protocol.

4.2.  SCIM Endpoints

   If the FastFed Identity Provider supports additional SCIM attributes
   beyond those mappable to OpenID Claims as per the table above, then
   the FastFed Identity Provider MUST implement the OpenID Connect SCIM
   Profile 1.0 [DRAFT] [OpenID.SCIMProfile].

   FastFed Application Providers MAY use the SCIM profile to access the
   full set of user information available.  Alternatively, they MAY use
   the OpenID Standard Claims to access only a subset of the available
   information.

4.3.  Privacy Considerations

   When exposing end-user information through OpenID Claims or SCIM
   endpoints, FastFed Identity Providers MUST not expose any attributes
   which were not in the "desired_user_attributes" element of the
   Application Provider Metadata (Section 3.3.4 of FastFed Core
   [FastFed.Core]).

5.  Interoperability Requirements

   Each identity standard defines a set of optional features to enable
   usage in a wide variety of circumstances.




McAdams                 Expires December 8, 2018                [Page 7]

                 FastFed Profile for OpenID Connect 1.0        June 2018


   However, a consequence of the flexibility is that two Providers may
   find themselves incompabitible despite sharing the same protocols.
   For example, if an Application Provider requires enrypted OpenID
   Connect ID Tokens, but the Identity Provider doesn't support this
   feature of OpenID Connect, the Providers are incompatible even though
   both use OpenID Connect.

   To deliver the simplified experience that is the goal of FastFed, it
   is important that two FastFed-enabled Providers have confidence that
   they can interoperate when sharing the same protocols.

   The following sections describe the subset of the OpenID Connect
   specifications that Providers MUST implement to be FastFed Combatible
   for the "OIDC" SSO protocol.  Providers MAY support additional
   functionality, but MUST NOT require the additional functionality when
   configuring federation with another Provider using FastFed
   specifications.

5.1.  Identity Provider Requirements

   o  MUST implement the required functionality of an OpenID Provider as
      defined in the OpenID Connect Core 1.0 [OpenID.Core]
      [OpenID.Core].

   o  MUST support the following flows:

      *  Authorization Code Flow (Section 3.1 of [OpenID.Core])

      *  Implicit Flow (Section 3.2 of [OpenID.Core])

      *  Hybrid Flow (Section 3.3 of [OpenID.Core]

   o  MUST support returning OIDC Claims through the UserInfo Response
      and the ID Token (Section 5 of [OpenID.Core]).

   o  MUST support the OIDC Client Authentication method
      "client_secret_basic" as defined in Section 9 of [OpenID.Core].

   o  MUST support ID Token signing using Asymetric Signatures with an
      "alg" value of "RS256", as defined in Section 10.1 of
      [OpenID.Core].

   o  MUST support Initiating Login from a Third Party, as defined in
      Section 4 of [OpenID.Core].

   o  MUST implement the required functionality of an OpenID Provider as
      defined in OpenID Connect Dynamic Client Registration 1.0
      [OpenID.Registration] [OpenID.Registration].



McAdams                 Expires December 8, 2018                [Page 8]

                 FastFed Profile for OpenID Connect 1.0        June 2018


   o  MUST implement the required functionality of an Open ID Provider
      as defined in Section 3 and Section 4 of OpenID Connect Discovery
      1.0 [OpenID.Discovery] [OpenID.Discovery]

      *  The property "registration_endpoint" in the OpenID Provider
         Metadata MUST contain the URL of the OP's Dynamic Client
         Registration Endpoint [OpenID.Registration].

      *  The property "token_endpoint_auth_methods_supported" in the
         OpenID Provider Metadata must contain the value
         "client_secret_basic".

   o  MUST specify all values in the OpenID Provider Configuration
      (Section 4 of [OpenID.Discovery] that are necessary to implement
      the functionality described in this section.

5.2.  Application Provider Requirements

   o  MUST implement the required functionality of an OpenID Relying
      Party as defined in the OpenID Connect Core 1.0 [OpenID.Core]
      [OpenID.Core].

   o  MAY use any of the following flows:

      *  Authorization Code Flow (Section 3.1 of [OpenID.Core])

      *  Implicit Flow (Section 3.2 of [OpenID.Core])

      *  Hybrid Flow (Section 3.3 of [OpenID.Core])

   o  MAY retrieve Claims via the UserInfo Response or ID Token
      (Section 5 of [OpenID.Core])

   o  MUST support the OIDC Client Authentication method
      "client_secret_basic" as defined in Section 9 of [OpenID.Core].

   o  MUST support ID Token signing using Asymetric Signatures with an
      "alg" value of "RS256", as defined in Section 10.1 of
      [OpenID.Core].

   o  MAY support Initiating Login from a Third Party, as defined in
      Section 4 of [OpenID.Core].

   o  MUST dynamically register with the Identity Provider when enabling
      federation during the FastFed Handshake.  To do so, the entity
      MUST implement the required functionality of an Open ID Client as
      defined in Section 3 of [OpenID.Registration].




McAdams                 Expires December 8, 2018                [Page 9]

   o  MAY use the information in the OpenID Provider Configuration
      [OpenID.Discovery] to decide whether to use additional OpenID
      capabilities, above and beyond the FastFed minimum interoperablity
      requirements.  The additional capabilities must be optional and
      the federation MUST still succeed if one Provider does not support
      the additional capabilities.

5.3.  Key Rotation

   Open ID Connect describes a variety of keys and secrets that are
   managed by both Identity Providers and Application Providers.  For
   reference, the FastFed Interoperability Requirements defined in this
   specification permit use of the following:

   o  Asymmetric Signing Keys as defined Section 10 of OpenID Connect
      Core 1.0 [OpenID.Core] [OpenID.Core], consisting of the following:

      *  A private key used by the Identity Provider to sign ID Tokens
         issued to the Application Provider.

      *  A public key used by the Application Provider to verify ID
         Token signatures.  The Identity Provider publishes the public
         keys in the location described by the "jwks_uri" in the
         Identity Provider's OpenID Connect Discovery [OpenID.Discovery]
         configuration.

   o  Values for "client_secret" and "client_secret_expires_at" that are
      vended from the Identity Provider to the Application Provider as
      described in Section 3.2 of OpenID Connect Dynamic Client
      Registration 1.0 [OpenID.Registration] [OpenID.Registration].
      FastFed Interoperability requires that "client_secret" be of type
      "client_secret_basic" as defined in the HTTP Basic Authentication
      Scheme specified in Section 2.3.1 of OAuth 2.0 [RFC6749].

   FastFed Providers MUST implement automatic key rotation.  This
   section describes the interoperability requirements for OpenID
   Connect key rotation.

5.3.1.  Key Rotation Requirements for an Identity Provider

5.3.1.1.  Publication of New Signing Keys

   To publish new signing keys, the following requirements apply to the
   Identity Provider:

   o  MUST add a new key to the response value from the "jwks_uri" at
      least 14 days before prior keys expire.




McAdams                 Expires December 8, 2018               [Page 10]

                 FastFed Profile for OpenID Connect 1.0        June 2018


   o  SHOULD wait at least 7 days after publishing a new key before
      using it for signing ID Tokens, to give consumers time to read the
      new key.  MAY continue using the previous key if problems arise
      with the new key, to give time for diagnosis of the problems.

   o  SHOULD stop using the older key for signing ID Tokens when less
      than 1 day remains before the key expires.

   o  MUST support HTTP 1.1 ETag semantics [RFC2616] for all requests to
      the "jwks_uri", including:

      *  Returning an ETag response header

      *  Accepting an If-None-Match request header, and returning an
         HTTP 304 response if the SAML Metadata is unmodified,

   o  MAY return an HTTP 301 response code to indicate the "jwks_uri"
      has moved to a new location.

5.3.1.2.  Publication of New Client Secret

   To publish a new client_secret, the following requirements apply to
   the Identity Provider:

   o  MUST return a new "client_secret" in the response to queries
      against the "registration_client_uri", as defined in Section 4 of
      OpenID Connect Dynamic Client Registration 1.0
      [OpenID.Registration] [OpenID.Registration].

   o  MUST make the new "client_secret" available for consumption at
      least 14 days before the prior "client_secret" is scheduled to
      expire.

   o  MUST continue accepting the previous "client_secret" until the
      expiration date.

   o  SHOULD alert the Administrator if the previous client_secret
      continues to be used and less than 7 days remains until
      expiration.  The alert mechanism is an implementation detail that
      is outside the scope of the specification.

   o  MUST support HTTP 1.1 ETag semantics [RFC2616] for all requests to
      the "registration_client_uri", including:

      *  Returning an ETag response header

      *  Accepting an If-None-Match request header, and returning an
         HTTP 304 response if the SAML Metadata is unmodified,



McAdams                 Expires December 8, 2018               [Page 11]

                 FastFed Profile for OpenID Connect 1.0        June 2018


   o  MAY return an HTTP 301 response code to indicate the
      "registration_client_uri" has moved to a new location.

5.3.2.  Key Rotation Requirements for an Application Provider

5.3.2.1.  Consumption of New Signing Keys

   To consume new public signing keys, the following requirements apply
   to the Application Provider:

   o  MAY re-retrieve keys on demand by following the specifications
      defined in Section 10.1.1 OpenID Connect Core 1.0 [OpenID.Core]
      [OpenID.Core].

   o  Alternatively, MAY preemptively re-retrieve keys by querying the
      "jwks_uri" on a recurring basis to check for updates.  If this
      approach is taken, the Application Provider MUST re-retrieve the
      keys at least once every 24 hours.  Because keys can be rotated at
      any time, the Application Provider MUST NOT wait for the
      expiration date to check for updated keys.

   At least one of the above methods MUST be used to re-retrieve keys
   from the "jwks_uri".  Both methods MAY be used concurrently.  In
   addition, the following requirements apply to the Application
   Provider:

   o  MUST include the If-None-Match header in the request to the
      "jwks_uri", populated with the value of the ETag response header
      received when previously querying the "jwks_uri".

   o  MUST handle an HTTP 304 response code which indicates that the
      content at the "jwks_uri" is unchanged.

   o  MUST handle an HTTP 301 response code and use the new location for
      all future queries against the "jwks_uri".

   o  MUST accept OIDC ID Tokens signed using any of the valid keys
      contained in the response from the "jwks_uri".

5.3.2.2.  Consumption of New Client Secret

   As background, when an Application Provider initially registers with
   an Identity Provider using OpenID Connect Dynamic Client Registration
   [OpenID.Registration], the registration response contains the
   following attributes:

   o  client_secret




McAdams                 Expires December 8, 2018               [Page 12]

                 FastFed Profile for OpenID Connect 1.0        June 2018


   o  client_secret_expires_at

   o  registration_client_uri

   To rotate the "client_secret", the Application Provider will
   periodically re-query the Client Configuration Endpoint (located at
   the "registration_client_uri") in order to re-read the current
   configuration.  If a new "client_secret" has been issued, the new
   "client_secret" will appear in the response.  The specifications for
   querying the Client Configuration Endpoint are described in Section 4
   of OpenID Connect Dynamic Client Registration 1.0
   [OpenID.Registration] [OpenID.Registration].

   To consume new values of "client_secret", the following requirements
   apply to the Application Provider:

   o  MUST re-retrieve the client_secret from the
      "registration_client_uri" at least once every 24 hours.  Because
      secrets can be rotated at any time, the Application Provider MUST
      NOT wait for the expiration date to check for updated secrets.

   o  MUST include the If-None-Match header in the request to the
      "registration_client_uri", populated with the value of the ETag
      response header received when previously querying the
      "registration_client_uri".

   o  MUST handle an HTTP 304 response code which indicates that the
      content at the "registration_client_uri" is unchanged.

   o  MUST handle an HTTP 301 response code and use the new location for
      all future queries against the "registration_client_uri".

   o  SHOULD use the new "client_secret" for OpenID Connect flows upon
      receiving an updated value.  MAY temporarily revert to using the
      prior "client_secret" if the prior secret has not expired and
      unexpected problems arise from the new "client_secret".  This
      allows time for diagnosis of problems.
