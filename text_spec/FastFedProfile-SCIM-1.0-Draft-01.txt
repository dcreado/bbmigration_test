   satisfy to describe themselves as FastFed Compatibile with SCIM user
   provisioning.

1.  Introduction

   This specification defines the functionality which a provider must
   satisfy to describe themselves as FastFed Compatible with SCIM user
   provisioning.

   It consists of the following extensions to the FastFed Core
   [FastFed.Core] specification:

   o  Additional metadata in the FastFed Handshake Flows in order to
      exchange SCIM provisioning endpoints.

   o  An interoperability profile describing the subset of the SCIM
      specifications which must be implemented in order to be FastFed
      Compatible for SCIM provisioning.

2.  FastFed Metadata

   Section 3.3.1 of the FastFed Core [FastFed.Core] specification
   defines the value ""SCIM"" as a valid provisioning mode.

   A Provider who includes ""SCIM"" as a supported mode MUST comply with
   the requirements of this specification.

3.  FastFed Handshake

   When using the SCIM provisioning protocol, the FastFed Identity
   Provider and Application Provider have various responsibilities to
   comply with the protocol.

   The Identity Provider fulfills the role of a SCIM Client.  It has a
   responsibility to provision user information into the Application
   Provider as described in Section 4.1.

   The Application Provider fulfills the role of a SCIM Service.  It has
   a responsibility to host a SCIM Endpoint and handle provisioning
   messages from the Identity Provider as described in described in
   Section 4.2

   This specification extends the FastFed Core [FastFed.Core] handshake
   messages with the additional attributes necessary for each party to
   fulfill their respective obligations under SCIM.

3.1.  FastFed Registration Response

   This specification extends the contents of the FastFed Registration
   Response (Section 6.2.3.3 of FastFord Core [FastFed.Core]) with the
   following attributes:

   scim_service_uri  REQUIRED.  Contains the URL of the Application
      Provider's SCIM Endpoint.

   If the SCIM Endpoint is access controlled, the Application Provider
   MAY also include a value for "oauth_token" as defined in
   Section 6.2.3.1 of FastFord Core [FastFed.Core].  If this value is
   provided, the Identity Provider MUST present the access_token as a
   Bearer Token when invoking the SCIM Endpoint, as specified in
   Section 2 of the SCIM Protocol Specification [RFC7644]

   The following is a non-normative example of the contents of a
   registration response message:

    {
     "scim_service_uri": "https://tenant-56789.app.example.com/scim",
     "oauth_token": {
       "access_token": "MTQ0NjJkZmQ5OTM2NDE1ZTZjNGZmZjI3",
       "token_type": "bearer",
       "refresh_token": "IwOGYzYTlmM2YxOTQ5MGE3YmNmMDFkNTVk",
       "expires_in": 3600
     }
    }

4.  Interoperability Requirements

   Each identity standard defines a set of optional features to enable
   usage in a wide variety of circumstances.

   However, a consequence of the flexibility is that two Providers may
   find themselves incompabitible despite sharing the same protocols.

   To deliver the simplified experience that is the goal of FastFed, it
   is important that two FastFed-enabled Providers have confidence that
   they can interoperate when sharing the same protocols.

   The following sections describe the subset of the SCIM Protocol
   specifications that Providers MUST implement to be FastFed Combatible
   for SCIM provisioning.  Providers MAY support additional
   functionality, but MUST NOT require the additional functionality when
   configuring federation with another Provider using the FastFed
   specifications.

4.1.  Identity Provider Requirements

   SCIM requirements for Identity Providers:

   o  MUST implement the required functionality of a SCIM Client as
      defined in [RFC7643] and [RFC7644].

   o  For each end-user that is authorized to access the Application:

      *  SHOULD replicate end-user information to the Application
         Provider within 60 minutes of an active end-user being created
         or updated.

         The Identity Provider MAY use any combination of the following
         operations to provision user information:

         +  Create account via _POST /Users_

         +  Update account details via _PUT /Users/{id}_

         +  Read list of accounts via _GET /Users_

         +  Read account details via _GET /Users/{id}_

         +  Filtering based on _"userName eq"_

      *  SHOULD replicate end-user deactivation to the Application
         within 15 minutes of the user being deactivated within the
         Identity Provider.

         The Identity Provider MAY use any combination of the following
         operations to deactivate users in the Application:

         +  Deactivate account via _PATCH /Users{id}_

         +  Read list of accounts via _GET /Users_

         +  Read account details via _GET /Users/{id}_

         +  Filtering based on _"userName eq"_

4.2.  Application Provider Requirements

   SCIM requirements for Application Providers:

   o  MUST implement the required functionality of a SCIM Service
      Provider as defined in [RFC7643] and [RFC7644].

   o  MUST support the following operations:

      *  Create account via _POST /Users_

      *  Update account details via _PUT /Users/{id}_

      *  Deactivate account via _PATCH /Users{id}_

      *  Read list of accounts via _GET /Users_

      *  Read account details via _GET /Users/{id}_

      *  Filtering based on _"userName eq"_

   o  SHOULD respond to account deactivation by revoking the ability for
      the end-user to use the Application, even if that end-user has an
      active session with an expiration date in the future.  The
      revocation mechanism is an implementation detail and outside the
      scope of this specification.  Revocation should occur within 15
      minutes of receiving the deactivation.
