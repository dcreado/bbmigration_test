            FastFed Profile for SCIM Provisioning - draft 01
                              fastfed-1_0

Abstract

   This specification defines the requirements to implement the FastFed
   Profile for SCIM Full Lifecycle provisioning.  This profile supports
   continual provisioning, update, and deprovisioning of end-users
   between the Identity Provider and Application Provider.

1.  Introduction

   This specification defines the functionality which a provider must
   implement to satisfy the FastFed Profile for SCIM Full Lifecycle
   provisioning.

   It consists of the following extensions to the FastFed Core
   [FastFed.Core] specification:

   o  Additional metadata in the FastFed Handshake Flows to exchange
      SCIM provisioning endpoints.

   o  An interoperability profile describing the subset of the SCIM
      specifications which must be implemented to be FastFed Compatible
      for SCIM Full Lifecycle provisioning.

1.1.  Requirements Notation and Conventions

   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
   "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and
   "OPTIONAL" in this document are to be interpreted as described in RFC
   2119 [RFC2119].

   In the .txt version of this specification, values are quoted to
   indicate that they are to be taken literally.  When using these
   values in protocol messages, the quotes MUST NOT be used as part of
   the value.  In the HTML version of this specification, values to be
   taken literally are indicated by the use of "this fixed-width font".

1.2.  Terminology

   This FastFed Profile uses the terminology defined in Section 1.2 of
   the FastFed Core [FastFed.Core] specification.

2.  FastFed Metadata

   This specification extends the FastFed Core [FastFed.Core] metadata
   (Section 3.3.1) with the following value for "provisioning_profiles":

   urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle  A
      Provider who includes this URN in their list of capabilities MUST
      comply with the requirements described in this specification.

   The following is a non-normative example of Provider Metadata showing
   the usage of the value:

 {
   "identity_provider": {
     "capabilities": {
       "provisioning_profiles": ["urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle"],
   ...
 }

3.  FastFed Handshake

   When using this SCIM provisioning profile, the FastFed Identity
   Provider and Application Provider have various responsibilities to
   comply with the protocol.





McAdams                 Expires February 28, 2020               [Page 2]

                  FastFed Profile for SCIM Provisioning      August 2019


   The Identity Provider fulfills the role of a SCIM Client.  It has a
   responsibility to provision user information into the Application
   Provider as described in Section 4.1.

   The Application Provider fulfills the role of a SCIM Service.  It has
   a responsibility to host a SCIM Endpoint and handle provisioning
   messages from the Identity Provider as described in described in
   Section 4.2

   This specification extends the FastFed Core [FastFed.Core] handshake
   messages with the additional attributes necessary for each party to
   fulfill their respective obligations under SCIM.

3.1.  FastFed Registration Request

   This specification extends the contents of the FastFed Registration
   Request (Section 7.2.3.1 of FastFord Core [FastFed.Core]) with a
   structure sharing the same name as the profile:
   ""urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle"".

   The structure contains the following attributes:

   provider_authentication  REQUIRED.  A structure containing the
      authentication keys necessary to authenticate the provisioning
      provider, as defined in Section 3.3.3 of FastFord Core
      [FastFed.Core].

      The authentication keys MAY be the same as those defined in the
      Identity Provider's FastFed Metadata.  Alternatively, SCIM
      provisioning may be delegated to a distinct sub-system which
      authenticates with its own key materials.  The declaration of
      protocol-specific authentication materials enables both scenarios.

   The following is a non-normative example of the contents of a
   registration request message:

    {
      "iss": "https://idp.example.com",

      "schema": "urn:ietf:params:scim:schemas:core:2.0:User",

3.2.  FastFed Registration Response

   This specification extends the contents of the FastFed Registration
   Response (Section 7.2.3.3 of FastFord Core [FastFed.Core]) with a
   structure sharing the same name as the profile:
   ""urn:ietf:params:fastfed:1.0:provisioning:SCIM:FullLifeCycle"".

   The structure contains the following attributes:

   scim_service_uri  REQUIRED.  Contains the URL of the Application
      Provider's SCIM Endpoint.

   oauth_metadata  REQUIRED.  A structure containing the OAuth metadata
      for authenticating to the SCIM endpoint, as defined in Section 6.6
      of FastFed Core [FastFed.Core].

      The OAuth scope returned in this structure MUST authorize the SCIM
      client to perform all provisioning activity specified by this
      profile.

   The following is a non-normative example of the contents of a
   registration response message:

4.  Interoperability Requirements

   Each identity standard defines a set of optional features to enable
   usage in a wide variety of circumstances.

   However, a consequence of the flexibility is that two Providers may
   find themselves incompabitible despite sharing the same protocols.

   To deliver the simplified experience that is the goal of FastFed, it
   is important that two FastFed-enabled Providers have confidence that
   they can interoperate when sharing the same protocols.

   The following sections describe the subset of the SCIM Protocol
   specifications that Providers MUST implement to be FastFed Compatible
   for this profile.  Providers MAY support additional functionality,
   but MUST NOT require the additional functionality when configuring
   federation with another Provider using the FastFed specifications.

4.1.  Identity Provider Requirements

   SCIM requirements for Identity Providers:

   o  MUST implement the required functionality of a SCIM Client as
      defined in [RFC7643] and [RFC7644].

   o  MUST authenticate to the SCIM endpoint using an OAuth
      "access_token".  The token must be attained using the mechanisms
      defined in Section 6.6 of the FastFord Core [FastFed.Core]
      specification.  The keyId used to attain the OAuth token must
      exist in the keys hosted at the "jwks_uri" specified by the SCIM
      "provider_authentication" in Section 3.1.

   o  For each end-user that is authorized to access the Application:

      *  SHOULD replicate end-user information to the Application
         Provider within 60 minutes of an active end-user being created
         or updated.



McAdams                 Expires February 28, 2020               [Page 5]

                  FastFed Profile for SCIM Provisioning      August 2019


         The Identity Provider MAY use any combination of the following
         operations to provision user information:

         +  Create account via _POST /Users_

         +  Update account details via _PUT /Users/{id}_

         +  Read list of accounts via _GET /Users_

         +  Read account details via _GET /Users/{id}_

         +  Filtering based on _"userName eq"_

      *  SHOULD replicate end-user deactivation to the Application
         within 15 minutes of the user being deactivated within the
         Identity Provider.

         The Identity Provider MAY use any combination of the following
         operations to deactivate users in the Application:

         +  Deactivate account via _PATCH /Users{id}_

         +  Read list of accounts via _GET /Users_

         +  Read account details via _GET /Users/{id}_

         +  Filtering based on _"userName eq"_

      *  SHOULD populuate the "groups" attribute on a SCIM User object
         with a list of groups to which the user belongs.  As specified
         in the SCIM Core Schema [RFC7643], a user can belong to a group
         either through direct membership, through nested groups, or
         dynamically calculated.

4.2.  Application Provider Requirements

   SCIM requirements for Application Providers:

   o  MUST implement the required functionality of a SCIM Service
      Provider as defined in [RFC7643] and [RFC7644].

   o  MUST support authentication via OAuth "access_tokens".

   o  MUST provide an OAuth authorization server capable of issuing
      tokens as defined in Section 6.6 of the FastFord Core
      [FastFed.Core] specification.  OAuth tokens MUST only be issued if
      the token request is signed using a keyId which exists in the key




McAdams                 Expires February 28, 2020               [Page 6]

                  FastFed Profile for SCIM Provisioning      August 2019


      set hosted at the "jwks_uri" specified by the SCIM
      "provider_authentication" in Section 3.1.

   o  MUST support the following operations:

      *  Create account via _POST /Users_

      *  Update account details via _PUT /Users/{id}_

      *  Deactivate account via _PATCH /Users{id}_

      *  Read list of accounts via _GET /Users_

      *  Read account details via _GET /Users/{id}_

      *  Filtering based on _"userName eq"_

   o  SHOULD respond to account deactivation by revoking the ability for
      the end-user to use the Application, even if that end-user has an
      active session with an expiration date in the future.  The
      revocation mechanism is an implementation detail and outside the
      scope of this specification.  Revocation should occur within 15
      minutes of receiving the deactivation.
